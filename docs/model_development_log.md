# 시뮬레이션 모델 개발 이력

## 1. 보행 모델 선정

### 1.1 GCFM (Generalized Centrifugal Force Model)
- **출처**: Chraibi et al. (2010)
- **선정 이유**: 타원형 몸체, 자연 대기행렬 형성, JuPedSim 내장
- **검증 결과 (V1~V7)**:
  - V1 자유보행: PASS (오차 3.3%)
  - V2 기본 다이어그램: PASS (RMSE=0.184)
  - V3 병목 유량: PASS (최대 9.5%, Seyfried 2009 기준 10% 이내)
  - V4 MNL 게이트 선택: PASS (χ² p=0.587)
  - **V5 dt 민감도: FAIL** — dt=0.01→0.005 변경 시 유량 24% 차이
  - **V6 겹침: FAIL** — 최대 22.4cm 겹침 (33,387건 중 153건, 0.46%)
  - V7 대칭: PASS

### 1.2 문제점 → CFSM V2 전환 결정
- **V5 FAIL 원인**: GCFM은 힘 기반 모델로, dt가 작을수록 힘 적분이 정밀해짐. dt=0.01에서 반발력 계산이 부정확 → 결과 불안정
- **V6 FAIL 원인**: max_neighbor_repulsion_force=3.0N 상한 → 고밀도에서 반발력 부족 → 에이전트 겹침. JuPedSim 기본값은 9.0N이나, 높이면 튕김 현상 발생
- **Q1.3 원논문 대조**: Chraibi (2010) 원논문에는 반발력 상한이 없음. JuPedSim 구현에서 추가된 제약

### 1.3 CFSM V2 (Collision-Free Speed Model V2)
- **출처**: Tordeux et al. (2016)
- **전환 이유**: 속도 기반 모델 → 충돌 원천 차단 (겹침 0), dt 민감도 낮음
- **캘리브레이션**: time_gap 파라미터를 Seyfried (2009) 병목 유량 기준으로 보정
  - 문헌값 1.06s → **0.80s**로 조정 (0.8m 병목에서 오차 1.2%)
- **검증 결과**:
  - V1 자유보행: PASS (오차 0.0%)
  - V2 기본 다이어그램: PASS (RMSE=0.134)
  - V3 병목 유량: PASS (0.60m: 6.6%, 0.80m: 1.2%, 1.00m: 2.7%)
  - V6 겹침: PASS (0건, 0.0cm)

## 2. 게이트 통과 메커니즘

### 2.1 물리적 배리어 접근 (실패)

**시도 1: 원래 폭 (0.55m 통로, 1.5m 두께)**
- 문제: CFSM 에이전트(반경 0.15m)가 좁은 통로에서 서로 끼임 (낑김)
- 원인: 게이트 간격 0.85m에 통로 0.55m → 측벽과 에이전트 간 기하 반발로 정체

**시도 2: 넓은 통로 (1.2m)**
- 문제: 통로(1.2m) > 게이트 간격(0.85m) → 인접 통로 겹침 → 벽 소멸 → 에이전트 자유 통과
- 원인: 물리적 벽 자체가 사라져 게이트 우회 가능

**시도 3: 얇은 배리어 (0.2m 두께, 0.70m 통로)**
- 문제: 에이전트가 통로 통과 시 측벽에 튕김
- 원인: 0.70m 통로 내에서 CFSM 기하 반발이 작동

**시도 4: 통로 폭 = 게이트 간격 (0.85m)**
- 문제: 클러스터 내부 측벽은 없어졌지만, 시각적 게이트 통로(0.55m) 밖으로 이동
- 원인: 물리적 제약이 없으니 시각적 통로를 벗어남

### 2.2 소프트웨어 제어 접근 (채택)

**최종 구조: 배리어 없음 + queue_stage + waypoint 유도**

```
계단 → [1차 선택] → approach_wp(10.0, gate_y) → queue_stage(대기) → [pop] → post_gate_wp → 출구
```

- **물리적 벽 없음**: CFSM 기하 반발/튕김 문제 원천 제거
- **queue_stage**: 서비스 완료 전까지 에이전트 통과 차단 (소프트웨어 게이트)
- **waypoint 유도**: post_gate_wp가 게이트 출구 방향 → 에이전트가 게이트 중앙을 직선 통과
- **처리량 제어**: 서비스 시간 모델이 담당 (물리적 통로 폭이 아님)

## 3. 대기행렬 행태 개선

### 3.1 queue_stage 고정 위치 진동 문제

**문제**: CFSM 에이전트가 queue 대기 위치에 서있을 때, 주변 에이전트와의 충돌 회피로 미세 진동 발생
- 후진 2회+ 에이전트: 50/77명 (65%)
- 방향전환 평균: 14.6회

**해결**: 대기 위치 도달 시 desired_speed=0 설정
- enqueued 상태 + 대기 위치(gate_y) ±0.3m 이내 → speed=0 (정지)
- 앞사람 pop → 나머지 에이전트 speed 복원 → 한 칸 전진 → 도달 → speed=0
- **결과**: 후진 2+ → 8명, 방향전환 평균 14.6 → 2.5회

### 3.2 중앙 몰림 현상

**문제**: 모든 에이전트가 기본 journey(G4, 중앙 게이트)로 먼저 이동 → 중앙에 모여서 꿈틀댐
- 원인: 1차 게이트 선택(dist=3.0m, x=9.0)이 approach_wp(x=8.0)보다 뒤에서 발동
- 에이전트가 G4 대기열에 먼저 진입 → 재배정 → 방향 전환

**해결**: 1차 선택 거리 확대 (3.0m → 9.0m)
- 계단(x≈3) 직후 게이트 선택 → 처음부터 해당 게이트로 직행
- approach_wp(x=10.0)를 1차 선택(x=3~4) 뒤에 배치 → 올바른 게이트로만 접근

### 3.3 과도한 재선택 (헤매는 행태)

**문제**: 3단계 재선택(Gao 2019 LRP)에서 2m 구간 내 최대 3번 게이트 변경
- 경로변경 51회/77명 (66%), 3차 재선택 18회
- 에이전트가 게이트 앞에서 좌우로 왔다갔다

**해결**:
1. **3차 재선택 제거**: 게이트 1m 앞에서 방향 전환 금지
2. **2차 재선택 조건 강화**: 현재 게이트 대기열 3명 이상 + 새 게이트가 확실히 짧을 때만
3. **queue 진입 후 재선택 금지**: enqueued 에이전트는 locked (게이트 확정)
- **결과**: 경로변경 51 → 2회, 3차 재선택 18 → 0회
